# 阶段 1 / 7 — 项目骨架与本地基础设施

## 上文衔接

这是商品管理系统的第一个开发阶段，目前没有任何已完成的代码。本阶段产出是后续所有功能开发的地基，必须严格按照下面的架构搭建，因为后续阶段不会重新调整目录结构。

## 全局架构约束（必须遵守）

采用 Clean Architecture 四层：

```
internal/domain/entity      纯业务实体，不依赖任何外部包
internal/domain/port        Repository / Storage / EventPublisher / Notifier 等接口定义
internal/usecase            业务用例编排，只依赖 domain/port 的接口
internal/interface/http     Gin handler、中间件、路由表
internal/infrastructure     具体技术实现（GORM、Redis、本地磁盘等）
```

domain 层和 usecase 层禁止 import 任何具体技术实现（gorm.io、aws-sdk-go 等）。所有外部依赖通过接口注入，依赖注入容器手写一个简单的 `internal/container/container.go` 即可，不需要引入 Wire 等代码生成工具（除非你认为有更好的轻量方案，可以说明理由后采用）。

技术栈：Go 1.22+、Gin、GORM、PostgreSQL、Redis、Viper（配置）、zap（日志）、Docker。

## 本阶段目标

只搭骨架，跑通基础设施和最简单的 health check 接口，**不需要实现任何业务功能**（用户、商品等下个阶段再做）。

## 具体任务清单

1. 初始化 Go module，创建上述目录结构（即使某些目录暂时是空的也要建好，体现完整骨架）
2. 实现配置加载：`internal/infrastructure/config/`，用 Viper 读取 `configs/config.local.yaml` 和环境变量（环境变量优先级高于文件，方便云上部署时覆盖），至少包含数据库连接信息、Redis 连接信息、JWT 密钥、服务端口这几类配置项
3. 实现日志初始化：用 zap，输出结构化 JSON 日志，本地开发模式可以用更易读的 console 格式（通过配置区分 dev/prod 两种日志格式）
4. 搭建 GORM 数据库连接的基础设施代码（`internal/infrastructure/persistence/gorm/`），连接池参数显式配置（MaxOpenConns 至少设到 50 以上，MaxIdleConns 合理设置，体现并发要求），但**暂时不需要任何具体的 Repository 实现**，建好连接初始化代码即可
5. 搭建 Redis 连接的基础设施代码（`internal/infrastructure/cache/redis/`），同样只需要连接初始化
6. 实现一个最简单的依赖注入容器 `internal/container/container.go`，负责初始化配置、日志、数据库连接、Redis 连接，并能优雅地把这些资源传递给后续层（哪怕现在还没有 usecase 可以注入，也要把容器结构搭好，方便后续阶段往里加）
7. 实现 Gin 路由骨架：`internal/interface/http/router/`，先约定路由注册的方式区分 `Public` 和 `Protected` 两类（具体的 RBAC 中间件下个阶段实现，这一步只需要先把注册接口的"形状"定义出来，比如 `router.Public(method, path, handler)` 这样的函数签名，哪怕内部实现暂时和 Protected 一样也没关系，占位即可）
8. 实现一个 health check 接口：`GET /health`，返回服务状态、数据库连接状态、Redis 连接状态（实际 ping 一下，不要只返回固定字符串），这是验证本阶段所有基础设施都接通的关键接口
9. 编写 `Dockerfile`（多阶段构建，最终镜像基于 alpine 或 scratch，体积尽量小）
10. 编写 `deployments/docker-compose.yml`，一键启动：app 容器 + PostgreSQL + Redis + MinIO（MinIO 先启动占位即可，本阶段不需要真正调用它）
11. 编写 `configs/config.example.yaml` 作为配置模板，并在 README 里说明如何复制成 `config.local.yaml` 并填入本地值
12. 编写最初版 README：项目简介、目录结构说明、本地启动步骤

## 占位说明

如果中途因为分层设计需要某个接口或某个目录提前出现（比如 domain/port 下预留 `Repository` 通用接口、`EventPublisher` 接口签名），可以先把接口定义写出来但内部不需要实现，用注释说明"将在阶段 X 实现"即可，目的是让整体骨架的"形状"在第一阶段就基本确定，后面只是往里填血肉。
