# 阶段 6 / 7 — 云存储实现与部署文件

## 上文衔接

阶段 1-5 已完成：分层骨架、用户认证、商品管理、文件上传（本地存储版本）、推送服务桩实现。本阶段把"本地存储"补全为可切换的云存储实现，并准备好部署到云端所需的所有文件。继续在已有目录结构里开发。

## 全局架构约束（必须遵守）

阶段 4 已经定义好了 `Storage` 接口，本阶段只是新增一个实现，**不应该改动这个接口的签名**，也不应该改动 usecase 层任何代码。如果发现接口设计在实际接 S3 时有缺陷需要调整，要明确说明改动原因，并确认对已有调用方影响最小。

## 本阶段目标

1. 实现 `Storage` 接口的 S3 兼容版本（同时适配 AWS S3 和阿里云 OSS，因为两者协议兼容）
2. 通过配置切换本地存储 / 云存储实现，不需要改代码，只改配置
3. 完善 Dockerfile、docker-compose、K8s 部署文件，为云端部署做好准备

## 具体任务清单

1. `infrastructure/storage/s3` 下实现 `Storage` 接口的 S3 兼容版本，使用官方或社区维护的 S3 SDK，配置项包括：endpoint（区分 AWS 默认 endpoint 和阿里云 OSS endpoint）、access key、secret key、bucket、region
2. 在依赖注入容器里，根据配置项 `storage.driver`（取值 `local` 或 `s3`）决定实例化哪个 `Storage` 实现，这是验证"接口隔离设计"是否真正生效的关键一步
3. 完善配置文件结构，把云厂商相关配置都通过环境变量注入（不要写死在代码或配置文件里），并在 `configs/config.example.yaml` 里补充这部分配置项的示例和注释
4. 检查并完善 Dockerfile：确认是多阶段构建，最终镜像基于 `scratch` 或 `alpine`，体积合理，且能在云端容器环境正常运行（注意 CGO_ENABLED=0 等交叉编译细节，确保本地 Mac/Windows 开发机构建出的镜像能在 Linux 容器环境跑起来）
5. 编写或完善 `deployments/k8s/` 下的 Kubernetes manifest（Deployment、Service、ConfigMap、Secret 模板），即使本次主要走容器服务（如 ECS）部署，也要把这套 K8s 文件准备好，作为未来可移植到任意云厂商 K8s 服务的备选方案
6. 在 README 里补充一节"切换云存储驱动"的说明：本地开发用 `storage.driver=local`，云端部署改成 `storage.driver=s3` 并填好对应环境变量即可，不需要改任何代码
7. 补充一份简要的"AWS 部署对照表"：本系统里哪些基础设施需要在 AWS 创建（ECR、RDS、ElastiCache、S3、ECS）、哪些配置项对应填什么
