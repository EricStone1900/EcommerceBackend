# 阶段 7 / 7 — 测试、压测与文档收尾

## 上文衔接

阶段 1-6 已经完成了系统的全部核心功能：分层架构骨架、用户认证与 RBAC、商品管理、文件上传与事件预留机制、推送服务、云存储实现与部署文件。本阶段是收尾阶段，目标是让系统达到"可以放心交付/上线"的状态，补齐测试、性能验证、文档。继续在已有目录结构里开发，本阶段不应该新增业务功能。

## 本阶段目标

1. 补充关键业务逻辑的单元测试
2. 提供压测脚本，验证 50+ 并发要求
3. 生成 API 文档（Swagger/OpenAPI）
4. 完善最终版 README，作为交付文档

## 具体任务清单

1. 为 `usecase` 层的核心用例补充单元测试，重点覆盖：注册登录的各种异常分支（密码错误、用户已存在等）、RBAC 权限判断逻辑、商品 CRUD 的边界情况、文件上传的类型/大小校验逻辑。测试中所有外部依赖（Repository、Storage、EventPublisher、Notifier）都要用 mock 实现（基于 domain/port 定义的接口生成 mock，可以用 `mockgen` 等工具自动生成），不需要真实连接数据库
2. 补充几个关键 HTTP 接口的集成测试（至少覆盖注册→登录→访问受保护接口这条核心链路），可以用真实的测试数据库（建议用 Docker 启动一个临时 PostgreSQL，或者用 sqlite 内存数据库做轻量集成测试，自行判断取舍）
3. 用 k6 或 vegeta 编写压测脚本（`scripts/loadtest.js` 或类似），至少覆盖：登录接口、商品列表接口、商品详情接口这几个高频路径，设置 50 并发、持续 1 分钟，输出响应时间分布（P50/P95/P99）和错误率
4. 本地用 docker-compose 跑起完整系统，执行一次压测，把结果（P95 延迟、错误率、有没有出现连接池耗尽等问题）记录下来，如果发现性能瓶颈（比如数据库连接池不够、缺少索引），在本阶段一并修复
5. 用 swaggo 或类似工具，给所有 handler 加上注释，自动生成 Swagger/OpenAPI 文档，挂载一个 `/swagger/index.html` 路径方便查看和调试
6. 编写最终版 README，至少包含：
   - 项目简介和整体架构图（文字描述即可，画图工具按需使用）
   - 本地启动步骤（从零开始，假设读者只有 Docker）
   - 环境变量完整说明表
   - 如何运行测试、如何运行压测
   - 如何切换本地存储/云存储
   - 部署到 AWS 的步骤指引（可以引用之前单独准备的 AWS 部署文档）
   - 未来接入 OCR 微服务、智能助手微服务、APNs 推送的具体指引（说明要改哪个文件、实现哪个接口，让接手的人不需要看代码就能知道怎么开始）
7. 检查整个项目，删除开发过程中遗留的临时调试代码、注释掉的废弃代码、明显的 TODO（确实需要保留的 TODO 要写清楚是什么、为什么留到以后）
